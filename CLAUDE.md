## Commands

All commands run from the repo root unless noted.

```bash
# Run all fast tests (no Gemini calls)
PYTHONPATH=backend pytest tests/ -m "not slow" -v

# Run a single test
PYTHONPATH=backend pytest tests/test_scorers.py::TestCalculateWcagRatio::test_negro_sobre_blanco_es_21 -v

# Run only unit tests (no credentials needed)
PYTHONPATH=backend pytest tests/ -m unit -v

# Run backend locally
cd backend && uvicorn main:app --reload --port 8000

# Verify syntax of all Python files
cd backend && python -m py_compile main.py models.py pipeline/step1_analyze.py pipeline/step2_evaluate.py pipeline/step3_generate.py pipeline/scorers/color_harmony.py pipeline/scorers/wcag_contrast.py pipeline/scorers/llm_scorers.py services/gemini_client.py
```

The backend requires `backend/.env` with `GOOGLE_API_KEY=...` to run locally. Copy `backend/.env.example`.

## Architecture

Quimera is a 3-step pipeline that **evaluates a design before generating code**. The pipeline runs as a FSM (finite state machine): each step enriches a single shared object (`DesignContext`) and passes it to the next.

```
User brief  -->  Step 1: analyze_and_design()   -->  DesignContext (colors, fonts, layout proposed)
                     |
                     v
             Step 2: aesthetic_evaluate()        -->  DesignContext (8 scores added)
                     |
              score >= 85?
              NO (iteration < 3): add critique, go back to Step 1
              YES: context.approved = True
                     |
                     v
             Step 3: generate_code()             -->  DesignContext (React component added)
```

### DesignContext is the single source of truth

`backend/models.py` defines `DesignContext` — a Pydantic model that accumulates state as it flows through the pipeline. Every pipeline function receives it, mutates it, and returns it. Nothing else is passed between steps.

### The 8 aesthetic criteria (Motor de Evaluacion Estetica)

Two types of scorers in `backend/pipeline/scorers/`:

- **Algorithmic (deterministic):** `wcag_contrast.py` (WCAG 2.1 formula, pure Python) and `color_harmony.py` (OKLCH color space, requires `colour-science`). These run synchronously.
- **LLM-based (temperature=0):** `llm_scorers.py` — 6 scorers that call Gemini 2.5 Pro. Designed to run concurrently via `asyncio.gather` in `step2_evaluate.py`. Temperature=0 makes them deterministic.

Overall score = average of all 8. Threshold = 85. Max 3 correction iterations before returning best result.

### Gemini SDK

Use `google-genai` (NOT the deprecated `google-generativeai`). All Gemini calls go through `backend/services/gemini_client.py`. The client uses:
- `model="pro"` -> `gemini-2.5-pro` (Steps 1 and 2)
- `model="flash"` -> `gemini-2.5-flash` (Step 3, code generation)

The Gemini SDK is synchronous; all calls are wrapped in `asyncio.to_thread()` to avoid blocking FastAPI's event loop.

### Test markers

- `@pytest.mark.slow` — calls Gemini (requires `GOOGLE_API_KEY`, counts against quota)
- `@pytest.mark.unit` — pure logic, no credentials needed
- Tests marked `skip` in `color_harmony.py` activate on Day 3 when `colour-science` OKLCH is integrated

### Deployment

- **Backend:** Docker image -> Artifact Registry -> Cloud Run (port 8080, `$PORT` env var)
- `backend/Dockerfile` uses `python:3.12-slim`
- `setup_gcp.sh` contains the 10-step GCP setup (project, APIs, Secret Manager, Cloud Run deploy, Cloud Build CI/CD)
- `GOOGLE_API_KEY` is stored in Secret Manager and injected as env var by Cloud Run
- `cloudbuild.yaml` (generated by `setup_gcp.sh`) triggers on push to `main`
